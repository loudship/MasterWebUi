## Target architecture and functional requirements


The target operating model should be implemented as two bounded backend services plus supporting infrastructure. The first service is a **FastAPI canonical model gateway** that owns the only approved contract for all model interactions. The second service is a **LangGraph orchestration service** that owns workflow state machines, tool routing, human approval, and response composition. The UI layer—whether Open WebUI, a React shell, or both—must become a thin client that talks only to those internal services and never directly to LM Studio, Qdrant, or crawler infrastructure. That separation addresses exactly what the audit called out: model-access duplication, internal monoliths, and mixed host/container assumptions. fileciteturn0file1

## Resource envelope and runtime constraints


Given your stated hardware envelope—an RTX 5070 with 12 GB of VRAM and 64 GB of system RAM—the system should be engineered around reliability-first admission control, not “best effort” concurrency. The infrastructure report is explicit that 12–16 GB class GPUs require aggressive quantization and careful offloading strategy for large models, especially when context windows grow or when multiple model workloads compete for the same VRAM. The coding agent should therefore assume that large dense models are not safe as the default interactive service class on this machine and that model loading, context length, and concurrent requests must be regulated centrally. fileciteturn0file0


The first hard runtime rule should be **one interactive generation model resident at a time**. LM Studio’s JIT loading, Idle TTL, and Auto-Evict behavior make this operationally feasible: TTL can unload idle models automatically, and Auto-Evict ensures that only one JIT-loaded model remains in memory by default. The gateway should exploit those features intentionally by assigning a bounded TTL to interactive requests and by refusing to expose multi-model client behavior directly to the rest of the platform. On this hardware, the safe starting policy is to make every generation request pass through a single-flight or queue-backed admission controller, with queueing preferred over VRAM exhaustion. citeturn26view0


The second hard rule should be **a separate embedding lane from the interactive chat lane**. LM Studio exposes embeddings through the same OpenAI-compatible server surface as chat and model listing, which means ungoverned embedding jobs can load or swap models on the same inference server that is serving interactive requests. On a 12 GB GPU, that is a direct path to model churn, latency spikes, or forced evictions. The blueprint should therefore require one of two patterns: either a compact embedding model that is CPU-friendly or low-VRAM and reserved for online query embeddings only, or a dedicated offline embedding window for bulk ingestion when the interactive generation lane is idle. Bulk re-embedding must never run concurrently with peak interactive chat traffic on this host. citeturn25view1turn25view2turn26view0 fileciteturn0file0


The third rule should be **bounded context by service class**. The infrastructure report discusses the tension between KV-cache residency, context length, and offloading on limited VRAM. The practical implementation consequence is that the gateway must define at least two request classes: a default interactive class and a long-context class. The default interactive class should start with a modest context ceiling—typically in the 8k to 16k range unless profiling proves a higher ceiling is safe for your chosen model—and the long-context class should be explicitly admitted, queued, and measured rather than silently accepted. This is the simplest way to keep “theoretical max context” from becoming “production outage by VRAM exhaustion.” fileciteturn0file0


The fourth rule should be **single-worker web services until persistence is hardened**. Open WebUI’s scaling guidance recommends `UVICORN_WORKERS=1` per container under orchestration and makes PostgreSQL, Redis, and an external vector database mandatory before multiple workers or replicas are safe. FastAPI’s concurrency model also distinguishes async I/O from CPU-bound or blocking workloads, which means the right pattern on a constrained single-node system is not “add workers early,” but rather “keep web layers async and thin, and move heavy work to controlled background lanes.” For this build, the initial gateway and orchestrator deployments should each start at one worker, with horizontal or internal scaling deferred until the persistence and air-gap gates pass. citeturn22view3turn18view2turn18view3


The fifth rule should be **strict control of background ingestion**. Crawl4AI’s self-hosting guide specifies `--shm-size=1g` for the container and shows `CacheMode.BYPASS` for fresh crawling, which matches the infrastructure report’s concern about stale extraction and Chromium stability. On this machine, the coding agent should enforce a concurrency policy of one crawl job at a time, low scheduling priority for crawl/index tasks, and explicit separation between interactive inference and browser-based extraction. If the system uses SearXNG offline search, its indexing path should also remain an internal maintenance workload rather than a continuously running high-churn background process. citeturn20view0turn20view3turn19view0 fileciteturn0file0

## Validation and deployment gates


No deployment phase should advance until its validation gate passes completely. That is especially important here because the repo audit was not a live runtime inspection and because the most consequential changes—air-gap enforcement, persistence migration, and gateway centralization—can appear “configured” while still failing in practice. The testing system should therefore be treated as part of the platform architecture, not as a postscript. LangGraph’s production testing guidance and FastAPI’s testing guidance provide a solid base for that approach. fileciteturn0file1 citeturn29view0turn18view0


The **unit gate** should cover every bounded module in isolation. For LangGraph, follow the framework’s own testing patterns: build graphs fresh inside tests, compile them with a per-test checkpointer, test individual nodes directly via `graph.nodes`, and use `update_state` plus `interrupt_after` to exercise partial execution paths. Add unit tests for gateway request normalization, model-policy enforcement, TTL injection, queue admission, secret fail-closed behavior, config reconciliation, and all repository adapters for PostgreSQL and Qdrant. ## Usability and maintenance standards


“Ease of use” in this environment should mean **reduced operator choice**, not more knobs. The coding agent should produce one source of truth for configuration, one generated runtime manifest, one documented bootstrap path, and one reconcile command that compares desired state with persisted state. That standard matters here because the repo audit already identified configuration drift and because Open WebUI explicitly marks many settings as `PersistentConfig` variables or otherwise persisted after first launch. In practice, that means the operator should not be editing several overlapping `.env` files and hoping they win; the platform should tell the operator what the effective state is. fileciteturn0file1 citeturn28view1turn28view2


The operator workflow should be intentionally small and scriptable. At minimum, the platform should ship with idempotent commands or task targets for `preflight`, `up`, `smoke`, `migrate`, `backup`, `restore`, `airgap-check`, and `status`. Each command should return machine-readable output for CI and should fail loudly on drift, secret absence, or dependency health issues. FastAPI lifespan hooks and Docker healthchecks should be used to make service readiness objective rather than anecdotal. citeturn18view1turn24view2


Modularity standards should directly correct the codebase problems identified in the audit. No service file should simultaneously own transport endpoints, business rules, persistence logic, and infrastructure utilities. The audit highlighted internal service files in the 700–1000 line range that mix exactly those concerns. The maintenance standard should therefore require bounded packages such as `gateway/`, `orchestrator/`, `persistence/`, `tooling/`, `security/`, and `ops/`, with repository adapters beneath them and contract tests around every cross-boundary interface. If a file or package boundary cannot be explained in one sentence, it is too broad. fileciteturn0file1


Security-related usability standards should also be formalized. If Open WebUI remains your user-facing shell, it should also use an explicitly managed `WEBUI_SECRET_KEY`, and if users authenticate through that surface in production, Redis-backed token revocation should be retained rather than omitted. Backup and restore should be routine, not exceptional.
